01 · WHAT

这个系统到底是干什么的？

medHarness2 是影像科报告的"阅卷老师 + 陪练教练"：医生写了一份 CT / 胸片 / MRI 报告，它负责打分、找错、给改进建议。

打个比方：把医生的报告当成一篇考场作文。系统先请几位 AI 陪练对着同一张片子各写一份报告当参照； 然后像老师阅卷一样，给每份报告体检（结构完不完整、内容全不全、写得好不好）； 再把医生的报告和 AI 的报告逐条对答案——你写了它没写？它写了你没写？位置写错了？—— 每处出入标上危害等级（写错左右和漏掉骨折，严重程度天差地别）； 最后汇总成排名、画像和错题本式的教学建议。

注意边界：这是一个pilot-only 的内部研究面板：当前 52 例是工程基线，另有 11 例纯 DMX 链路作为探索性证据。 它可以评估报告、比较候选并生成教学草稿，但还不是临床决策工具，也不代表正式验证结论。 本页所有数字来自真实 run 产物（构建时注入），页面会区分证据等级，不把探索性结果写成临床结论。

03 · PHILOSOPHY

分层设计哲学：从数据契约到研究证据

整个代码库只有一个组织原则：上层管流程，下层干专活，层与层之间只传带契约的数据。 这不是为了工程好看——是为了科研：契约、迁移、评估、审计和图表可以分别替换、单测和做消融。

工作流
Workflow
相当于门诊流程单。决定先做什么后做什么、结果存到哪。全系统只有这一层被允许写文件——所以任何产物都能追溯到是哪条流程写的。工作流只负责编排和落盘，模块负责业务组合，工具负责单一判断，基础设施负责模型、配置与审计。
模块
Module
相当于诊室。只有两间：「单份报告体检室」（打分+拆发现+查结构一条龙）和「两份报告对比室」（对答案+定危害一条龙）。工作流不直接摆弄工具，都是把病例交给诊室。
工具
Tool ×13
相当于检查仪器。确定性工具负责可复现的主结果；LLM 工具负责需要语言判断的环节，并保留模型、配置、原文证据和哈希指纹。两类工具可以独立替换，但 LLM 结果要经过稳定性和临床校准后才能进入正式研究。
基础设施
Infra
相当于水电网。一个统一的 LLM 调用口子（LLMClient）：工具说"我要问一次大模型"，至于问的是外部 API（DMX 代理的 gpt-5.5）、本地 HF 模型、还是 mock 假人，由配置文件决定，工具代码一行不用改。这就是"一套代码，多档火力"的开关。
04 · WORKFLOWS

核心工作流与研究闭环

规模逐级放大：单病例评估 → 批量读者评估 → 科室聚合 → 教学建议；正式研究结论还要经过临床标注与实验门禁。

五阶段闭环：从输入到可治理结论

1

契约与迁移

冻结 case、finding、report、hazard 和 run schema，旧产物先迁移、再验证。

2

生成与评估基准

按模态/部位生成候选，区分 formal_fresh、exploratory_fresh、artifact 和 debug_fallback。

3

重评估与实验

确定性主结果与 LLM 审计分开保存，重测稳定性、临床标注和六项实验门禁同步推进。

4

图表与控制面板

只从真实产物构建页面，显示 Git、来源健康、证据等级和未通过门禁。

5

临床与治理

双读者/仲裁、隐私和凭据治理完成后，才把探索性工程证据升级为正式研究结论。
