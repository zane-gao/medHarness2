# medHarness2 Web 项目状态面板收尾设计

## 1. 背景

`web/` 已完成一部分 2026-07-11 项目证据接入，包括纯 DMX 11 例评估、重测稳定性、OCR 完整性审计、生成基准、九条战线、六项实验门禁和 pilot10 标注进度。然而页面基础叙事仍停留在旧版本：只描述四条工作流、声称系统不做生成、把工具描述为完全确定性的纯函数，并引用旧分支、旧提交和旧测试数量。

本次收尾面向项目负责人本人，目标是维持原版视觉和章节顺序，同时准确回答：项目现在能做什么、哪些证据可信、当前主要问题是什么、下一步应做什么。

## 2. 已确认决策

- 采用“A：原顺序补齐”，不重构为状态优先首页或双栏工作台。
- 保留暖白背景、细边框、蓝绿状态色、顶部锚点导航、长页阅读、卡片与展开详情。
- 页面继续使用本地真实运行产物，不新增外部 API 调用。
- 生成页面与运行数据允许原样提交到远程仓库，无需脱敏。
- `web/index.html`、`web/control_panel.html`、`web/legacy/index.html` 和 `web/legacy/control_panel.html` 不再被 `.gitignore` 排除；实际存在的页面随本次更新提交。
- `web/index.html` 是本次完整更新的主状态面板；`web/control_panel.html` 是按最新运行产物重建的次级工程控制面板；`web/legacy/` 下的页面只作为历史快照保留并上传，不继续扩展其旧版实现。
- 上述授权不包括 API 密钥、访问令牌、私钥或密码；凭据仍不得进入仓库。
- 直接更新主分支，不创建 PR。

## 3. 信息架构

页面章节顺序保持不变：

1. 这是什么
2. 整体流程
3. 设计哲学
4. 工作流
5. 工具箱
6. 运行证据
7. 最新进展
8. 进度地图
9. 问题与下一步

页面不增加侧栏、标签页或新的交互范式。所有收尾改动复用现有组件、颜色和响应式规则。

## 4. 分章节更新

### 4.1 这是什么

- 将系统定位更新为“报告生成与评估研究工具链”，覆盖样本准备、报告生成、质量评估、错误审计、实验与教学建议。
- 删除“本系统不做报告生成”的过期结论。
- 明确当前是 `pilot-only` 工程研究基础设施，不是临床决策系统，也不产生临床有效性结论。

### 4.2 整体流程

- 保留现有 7 步主流程及输入/输出节点。
- 在现有步骤中补充数据契约、证据等级、质量门禁、checkpoint、主结果与 LLM 审计的关系。
- 明确生成模型按模态与部位路由，`formal_fresh`、`exploratory_fresh`、`artifact`、`debug_fallback` 和 `mock` 不得混用。

### 4.3 设计哲学

- 保留工作流、模块、工具、基础设施四层视觉结构。
- 删除“所有工具都是纯函数、同输入必然同输出、只有工作流能写文件”等不准确表述。
- 区分确定性主结果、非确定性 LLM 判断、hash-linked 审计、契约校验、checkpoint 与运行注册。
- 将基础设施说明更新为配置驱动的本地模型、DMX、显式备份路由和隐私策略。

### 4.4 工作流

- 保留 WF1 单病例、WF2 批量、WF3 科室聚合、WF4 教学建议四张主卡。
- 在同一章节追加紧凑的研究流程区，覆盖：schema 导出/迁移、benchmark 计划与执行、重评估、实验聚合、图表生成、临床标注和控制面板。
- 所有状态以当前代码和产物为依据，不把模板级能力写成已验证能力。

### 4.5 工具箱

- 保留 T1–T12 与质量门禁的原卡片结构。
- 更新 T1、T2、T4、T5、T6 的实现说明：严格 LLM judge、确定性候选或主结果、独立 reviewer、adjudication、hash-linked audit。
- 更新 T8 的证据等级和正式基准限制。
- 修正过期模型名、供应商状态和“失败自动静默降级”等描述。

### 4.6 运行证据

- 52 例 qualityfix run 继续作为工程基线展示。
- 明确该 run 含 14 份 artifact、67 份 debug fallback、0 份 formal fresh，不能作为正式模型排名。
- 阅读者、病例和报告明细可以保留在生成页面中并提交远程，符合用户本次明确授权。

### 4.7 最新进展

- 保留纯 DMX 11 例、三法官仲裁、重测稳定性、盲写生成、OCR 审计和 formal readiness 六类证据卡。
- 明确 11/11、99 份真实角色证据、零 fallback 是探索性工程证据，不是临床验证。
- 明确单次 LLM 评价重测不稳定；正式实验必须使用冻结聚合规则、不确定度报告和医生金标准校准。
- 明确旧 OCR 中 6 例确定截断、7 例疑似截断，均影响后续证据可信度。

### 4.8 进度地图

- 更新项目账本日期、主分支、HEAD、工作区状态、测试数量和控制面板状态。
- 保留九条战线、六实验门禁矩阵和 pilot10 进度。
- 控制面板只有在构建、测试和浏览器检查全部通过后才标记为工程层面的 `validated`。

### 4.9 问题与下一步

- 当前最高优先级依次为：临床金标准未开始、OCR 截断与基准未冻结、LLM 重测不稳定、52 例无 formal fresh、六实验无 validated 结果。
- 删除已成为历史状态的 DMX 403 主阻塞描述。
- 下一步必须与 `docs/project_status.yaml` 的 `next_gate` 一致。

## 5. 数据来源与构建

构建脚本只读取本地文件：

- Git：当前分支、HEAD、工作区状态。
- `docs/project_status.yaml`：阶段、发布就绪度、九条战线及下一道门禁。
- 52 例 run：`run_summary.json`、analysis CSV、病例评估文件。
- 11 例纯 DMX run：evaluation summary、case evaluations、stability summary。
- 生成基准：plan、benchmark summary 和结果 JSONL。
- OCR：完整性审计 summary。
- 实验：六项实验聚合 results。
- 标注：pilot10 manifest。

构建结果仍为自包含静态 HTML。模板只包含占位符和渲染逻辑；生成页包含完整注入数据，并根据用户授权纳入 Git。

主状态页由 `web/build_panel.py` 和 `web/panel_template.html` 维护。次级控制面板继续由 `medharness2 dashboard build` 及包内模板生成。legacy 构建脚本和模板保持可运行或可查阅，但不承担最新项目状态表达。

## 6. 错误处理

- 缺少默认 52 例 run、模板占位符或核心分析文件时构建失败并返回非零状态。
- 可选证据缺失时，页面显示“尚未产出”与预期来源，不再静默隐藏造成误解。
- 数字转换、空列表和零分母必须有明确保护。
- 构建完成后校验所有必需章节锚点和注入数据键。

## 7. 测试与视觉验证

- 为 Git 元数据、状态账本、可选证据缺失和聚合计数补充单元测试。
- 静态检查页面锚点、过期文案、硬编码病例号、模板占位符与生成页面跟踪状态。
- 使用真实本地产物重新构建全部生成页面。
- 使用 Playwright 检查桌面和移动视口、浏览器控制台、锚点导航、折叠详情、表格与矩阵溢出。
- 运行完整 `make test`。
- 提交前扫描常见令牌、私钥和嵌入式凭据；临床运行数据不作为阻塞项。

## 8. 验收标准

- 原版视觉和章节顺序保持不变。
- 页面不再包含已知过期事实和旧项目元数据。
- 52 例基线、11 例探索性证据、OCR 风险、标注状态和实验门禁相互区分。
- 页面明确显示 `pilot-only`，不暗示临床有效性。
- 所有必要数据缺失均可见或导致构建失败。
- 桌面和移动端无明显重叠、截断或横向溢出。
- `web/` 中代码、模板和生成页面均提交到主分支并推送远程。
- 完整测试通过，远程主分支哈希与本地一致。

## 9. 不在范围内

- 重新设计品牌或视觉语言。
- 增加后端 Web 服务、登录、数据库或实时任务控制。
- 补跑高成本模型、OCR 或临床实验。
- 生成新的医学结论、模型排名或论文级统计。
