# medHarness2 盲区扫描人话版（2026-07-24）

这份文档把 `blindspot_audit_20260714.md` 中的全部 37 条实际问题翻成日常语言，并按当前代码和最近一次回归重新标记状态。它不是新的审计，也不替代原始技术记录；原始文档保留了代码位置、验证方法和历史证据。

## 先看结论

大多数“程序会悄悄算错、把假数据当真数据、或者悄悄吞掉失败”的工程问题已经修好，并经过完整回归验证。现在最重要的剩余问题不是“程序会不会报错”，而是下面四件事：

1. 很多模型可以跑出文字，但还没有足够真实病例和医生复核证明文字可靠。
2. 外部评委、OCR 和云端模型的一些正式重复实验还没有在冻结病例集上完成。
3. 网页上的“取消任务”目前只改任务状态，不能保证已经运行的模型立刻停下来。
4. CT/MRI 的真实序列选择、结构模板和跨医院数据仍需要继续用真实病例验证。

## 状态说明

- **已修复**：原来的程序错误已改掉，并有测试覆盖。
- **部分修复**：程序已经有防线，但还缺真实外部调用、真实病例或人工复核。
- **仍需处理**：这件事现在仍可能影响使用结果。
- **按要求暂缓**：你此前要求安全问题不作为本轮开发阻塞项；这里仍解释其含义，但不把它列入当前开发待办。
- **不构成缺陷**：审计时怀疑过，后来确认不是实际问题。

## 一、原始 Critical 问题（1 条）

### C1. API 没有登录和文件范围限制

**人话：** 如果把 API 直接暴露给别人，调用者可以让服务器读取或写入不应该碰的文件，也可以反复触发模型任务。

**当前状态：按要求暂缓。** 这是安全问题，不影响这轮报告路由和融合功能，但不应把当前 API 当成可直接对外开放的正式服务。

## 二、原始 High 问题（17 条）

### H1. 原始病历可能直接发给外部服务

**人话：** 报告里的姓名、住院号、医生名等内容，过去可能原样发送给外部模型服务。

**当前状态：按要求暂缓。** 这是安全和隐私问题；功能上不阻塞本轮，但真实病人数据不能因为系统能跑就默认可以外发。

### H2. 强配置曾主动关闭隐私检查

**人话：** 为了让外部模型能正常调用，某些强配置把“先检查再外发”的门关掉了。这样虽然能跑，但不能证明没有外发敏感内容。

**当前状态：按要求暂缓。**

### H3. 当时没有“既能真实评测、又能安全处理文本”的完整模式

**人话：** 以前打开隐私门会让真实评测直接失败，关掉门又会把原始内容直接送出去，像“要么不能开车，要么不系安全带”。

**当前状态：按要求暂缓。**

### H4. 共享盘上的报告和凭据权限过宽

**人话：** 同一台机器上的其他账号有机会读到不该读的报告或配置内容。

**当前状态：按要求暂缓。** 这里不重复任何凭据内容。

### H5. 过去的排序会把很差的分数算得像很好

**人话：** 1 到 5 分的评分换算写错过，最差的 `1` 分在一个边界情况下会被当成满分；整体换算也会把分差压扁。

**当前状态：已修复。** 现在不再用那套错误换算。

### H6. 两个分数几乎一样时，系统过去硬选一个“赢家”

**人话：** 例如 A 是 72 分、B 是 71 分，但这 1 分可能只是抽样波动。旧逻辑会把 B 直接踢掉，看起来像 A 明显更好。

**当前状态：部分修复。** 系统会保留卡在分界线附近的候选并标记“需要复核”；真正要知道差异是否可信，仍要在冻结真实病例集上算不确定性。

### H7. 极少病例时的误差范围以前会骗人

**人话：** 只有 1 个病例时，旧系统还会给出一个看起来很精确的误差范围，其实根本没有统计意义。

**当前状态：已修复。** 单病例现在明确没有可信区间；小样本用更保守的算法。以后看报告时仍要确认展示层没有把“没有区间”误读成“没有误差”。

### H8. 过去的正式汇总没有做“多模型比较要扣分”的统计处理

**人话：** 同时比较很多模型时，总会偶然出现一个看起来最好。过去缺少必要的统计校正，容易把偶然领先当真领先。

**当前状态：部分修复。** 代码已经加入比较和校正；还没有足够的冻结真实结果去真正跑出可信结论。

### H9. 模拟评委过去可能被当成真实评委

**人话：** 本地的假评委会按固定规则给分，甚至“字数多、有两个标题”的垃圾报告也可能拿高分。过去这些分数有机会看起来像真实模型评出来的。

**当前状态：已修复主要风险。** 模拟和回退结果都会打标记，正式统计会排除它们。仍要记住：默认配置就是给开发调试用的 mock，不能拿默认运行结果写医学结论。

### H10. 汇总时过去可能把假分和真分平均在一起

**人话：** 即使一条结果写着“这是临时回退”，旧汇总也可能只拿数字平均，最后真假混成一个均值。

**当前状态：已修复主要风险。** 汇总会过滤 mock/fallback 来源；非正式流程仍可能产生调试结果，所以不能自动升级为正式 benchmark。

### H11. 过去会奖励“写得多”，不一定奖励“写得对”

**人话：** 旧指标按提到多少类问题打分。报告越长、哪怕多写了幻觉，也可能更高；简短但准确的报告反而吃亏。

**当前状态：已修复。** 这类模板覆盖率只保留作抽取诊断，不再拿来决定模型排名。

### H12. 同一个云模型重复跑，过去可能无法复现

**人话：** 温度设为 0 不等于每次都完全一样；以前请求没有稳定种子，重复跑可能得到不同答案，却没有办法追查原因。

**当前状态：部分修复。** 种子和配置指纹已经传递并记录，但外部服务是否真的完全遵守，还需要真实重复调用验证。

### H13. 用缓存重跑不等于独立重复实验

**人话：** 直接拿上次缓存的答案再算一遍，只能证明“读取缓存没坏”，不能证明模型第二次调用也会给相近答案。

**当前状态：部分修复。** 现在能强制不使用缓存重跑；真正的独立重复实验和差异报告还没完成。

### H14. 主评委和复核评委过去温度不一致

**人话：** 两个评委一个很稳定、一个很随机时，它们意见不一致到底是医学判断不同，还是参数不同，分不清。

**当前状态：部分修复。** 强配置已经明确 reviewer 的温度和种子；还要真实调用云端确认该模型确实支持这些参数，并做重复一致性测试。

### H15. 外部模型返回“夹杂解释文字的 JSON”时，过去容易解析失败

**人话：** 模型本来该回一段机器可读数据，但常会在前后加一句解释。旧程序只认非常规整的格式，容易把可用结果当失败。

**当前状态：已修复。** 现在会尝试从回答中找出 JSON；仍需要真实模型 smoke 持续验证各种返回格式。

### H16. 过去没有真正测试影像模态识别是否会认错

**人话：** 之前缺少测试来证明系统真的能区分胸片、CT、MRI，曾经有“只要是 PNG 就当胸片”的风险。

**当前状态：已修复。** DICOM 头信息、可信文件名、contact sheet 和未知输入都有专门测试；无法判断时会返回 `unknown`，不再硬猜胸片。

### H17. 同一个病灶换个写法，过去可能对不上

**人话：** 例如大小写、空格或连字符不同，本来同一个发现可能被当成两个不同发现，导致对齐和评分出错。

**当前状态：部分修复。** 现在会统一成稳定写法；但“肝囊肿”和“肝脏囊性灶”这种真正的同义表达，仍需要医学词表或医生确认，不能只靠字符串规则。

## 三、原始 Medium 问题（13 条）

### M1. “有几个模型”曾被误当成医学指标

**人话：** 系统曾可能把“这个病例跑了 3 个模型”也算成一个平均值，和真正的质量分摆在一起。

**当前状态：已修复。**

### M2. 百分位排名以前会把自己也算进去

**人话：** 自己和自己比较会让小样本排名偏高一点，像考试排名时把自己多算了一票。

**当前状态：已修复。**

### M3. 任何数字以前都可能被塞进统计表

**人话：** 只要一列看起来是数字，旧统计就可能把它当成绩，连不该比较的计数也混进去。

**当前状态：已修复。** 现在只统计明确允许的指标。

### M4. 程序 bug 曾被伪装成“模型临时失败”

**人话：** 真正的代码错误有时被大范围捕获，表面看像外部模型偶尔坏了，开发者会找错地方。

**当前状态：已修复。** 常见外部错误和程序错误现在分开处理。

### M5. 被评报告可能影响评委的指令理解

**人话：** 报告正文如果夹带“请给我满分”之类文字，评委模型可能被带偏；更常见的是评委解释写得像真的，其实没充分依据报告。

**当前状态：部分修复。** 输入已按不可信文本处理，也会标记解释是否缺少报告词语支撑；但不能自动判断医学解释是否真的合理，仍需真实评委和冻结病例验证。

### M6. 云服务限流和重试的真实表现还没充分验证

**人话：** 服务说“稍后再试”时，旧程序可能马上重试或处理得不够好，导致无谓失败。现在会识别常见限流信号，但不知道大规模真实运行时到底会失败多少。

**当前状态：部分修复。** 以后正式跑实验时必须同时报告重试次数、失败数和失败原因。

### M7. 过去一个病例只让评委判断一次

**人话：** 同一个评委可能这次说 4 分、下次说 3 分；只问一次就把它当真，会把偶然波动误当结论。

**当前状态：部分修复。** 已支持重复和一致性检查，但还没有完成真实冻结集上的重复评委实验。

### M8. 默认配置本来就是模拟模式

**人话：** 不指定强配置时，系统为了让开发方便会使用 mock。它能测试流程，但不是在调用真实医学模型。

**当前状态：仍需使用习惯配合。** 这是有意保留的开发能力，不是要删除的功能；正式运行必须显式选择真实配置，并检查结果里的来源标记。

### M9. 只看成功病例会显得成绩太好

**人话：** 失败病例如果被直接丢掉，最后平均分只剩“顺利完成的病例”，会比真实整体表现好看。

**当前状态：已修复统计出口。** 结果会保留总病例数、成功数和失败率；使用报告的人仍要同时看失败率，不能只看成功病例均分。

### M10. 没有真实评委时会用固定模板做风险判断

**人话：** 外部评委不可用时，系统会给出一份固定规则的风险提示。它能帮助调试，但不是医生判断。

**当前状态：已修复标记问题。** 这类结果会明确写成 deterministic fallback，并排除在正式统计之外；模板本身依然不能当临床证据。

### M11. 教学建议过去分不清是真模型写的还是模板写的

**人话：** 用户可能以为一段教学建议来自强模型，实际上它可能只是本地固定模板。

**当前状态：已修复来源标记。** 结果会区分 `llm_judge`、`mock_judge` 和 `deterministic_fallback`；真实 provider 的教学 smoke 仍需完成。

### M12. CI 测试通过不代表真实云模型能用

**人话：** 自动测试大多用假的网络和假的模型回答，所以代码绿灯只能说明程序逻辑没坏，不能证明真实云端接口、模型 ID、网络和 JSON 输出都正常。

**当前状态：部分修复。** 已有真实 `live-smoke` 门禁。Yunwu 报告生成已做过工程 smoke，但评委、OCR 等正式外部路径仍需要在可复现配置下完成真实调用和留证。

### M13. mock 生成的假报告过去还能参加打分

**人话：** 本地为了演示流程生成的占位报告，过去可能也进入排名，让结果看起来像模型真的写了报告。

**当前状态：已修复。** mock 报告只保留作诊断，不能进入正式排名和模型汇总。

## 四、原始 Low 问题（6 条）

### L1. 占位抽取器的警告不够显眼

**人话：** 系统曾从非常粗糙的占位抽取器开始，再让 LLM 修正。警告没有完全丢掉，但被放到了较深的元数据里，普通查看者不容易注意。

**当前状态：影响较低。** 追溯信息仍在；以后前端可以把这类“来源较弱”的提示展示得更醒目。

### L2. 一致率会混入不同模型和不同温度的差异

**人话：** 两个评委意见不同，可能因为医学判断不同，也可能只是一个更随机。这个噪声会让一致率稍微不纯。

**当前状态：影响较低，已缓解。** 参数已尽量统一；真正的稳定性仍要靠重复实测。

### L3. 过去汇总里的 fallback 数量被写死为 0

**人话：** 即使出现了回退，汇总也可能显示“没有回退”。

**当前状态：已修复。** 现在从实际结果统计。

### L4. 旧数据缺少关键观察内容时，过去会假装它是正常 finding

**人话：** 老格式里有些发现没有真正写清“看到了什么”，旧迁移会把它包装成正常发现，后续对齐可能被骗。

**当前状态：已修复。** 现在明确标成“旧数据没解析出来”，并禁止拿它参与真实 finding 配对。

### L5. 一处看代码，一处看原文，曾可能对同一发现得出不同结论

**人话：** 对齐器和风险评委以前优先看不同字段，同一个发现可能一边认为相同、另一边认为不同。

**当前状态：已修复。** 现在统一优先使用规范化字段。

### L6. 还保留了一些旧版本兼容代码

**人话：** 代码里有少量老字段的兜底分支，但当前严格合同下正常数据不会走到它们。

**当前状态：不影响结果。** 属于以后可做的清理，不是功能风险。

## 五、报告生成专项后来发现的盲区

这一部分不是把原始 37 条重复一遍，而是针对你要求的“多模型并行报告、Top-K、融合报告”后来专门核对的事项。

### R1. “任务完成”不一定等于“真的有报告”

**人话：** 以前系统可能只显示流程结束，却没有一份能看的候选报告。现在每个模型的空输出、异常和结构化失败都会单独列为 `candidate_failures`，不会再把“没有报告”藏起来。

**当前状态：工程上已修复。** 模型仍可能失败，但用户能知道是哪个模型、为什么失败。

### R2. 模态和部位匹配正确，不等于报告医学内容正确

**人话：** 系统可以知道“这是脑 MRI”，但模型仍可能把脑部问题写错，甚至编出不存在的病灶。路由只能减少“找错模型”，不能证明“模型写对了”。

**当前状态：仍需处理。** `BrainGemma3D` 已实测出现幻觉，因此被质量门拦住；任何未完成临床质量验证的模型都只能作为候选，不能被自动当最终诊断。

### R3. CT/MRI 不能直接喂给普通图片模型

**人话：** 云端通用视觉模型通常只看普通图片，而 CT/MRI 是一整套切片。系统现在会生成预览图，但预览图可能没有展示最关键的那几层。

**当前状态：部分修复。** 工程上已经能生成和校验 preview/contact sheet；真实复杂检查的序列选择和临床信息损失仍要继续验证。

### R4. CT/MRI 可能选到不合适的序列

**人话：** 一次 MRI 里可能有定位片、FLAIR、T1、T2 等很多序列。即使文件都合法，选错序列也会让模型看错东西。

**当前状态：部分修复。** 已有脑 MRI 优先 FLAIR、没有 FLAIR 时优先 T2 等规则和回归测试；不同医院的命名、复杂多序列检查和其他身体部位还不能保证都选对。

### R5. “取消任务”目前不等于“停掉正在跑的模型”

**人话：** 控制面板的取消按钮目前主要把数据库里的任务改成“已取消”。它还没有把这个 run 和实际运行中的 Python/模型进程绑定起来，所以后台任务可能继续跑到结束。

**当前状态：仍需处理。** 本地 legacy 子进程在超时、异常等场景已经会被独立进程组清理；但“用户在网页点取消”还没有真正向当前 worker 发停止信号。

### R6. 多模型并行后，最终结果仍可能等最慢的一个

**人话：** Yunwu 和本地模型现在会并行开始，本地 GPU 也按设备限流；但系统会等待本轮候选都返回后再统一排名和融合。一个特别慢或卡住的本地模型会拉长整例等待时间。

**当前状态：部分修复。** 已有并发、设备信号量、超时和失败记录；还缺“先把已完成的 Top-K 流式返回、慢模型后台补齐”之类的产品能力。

### R7. 两个病例用了同一个输出文件名，仍可能互相覆盖

**人话：** 批量合并会拒绝重复 `case_id`，但单病例接口允许调用方指定输出路径；如果两个人恰好写同一个文件，后写的会覆盖先写的。

**当前状态：仍需处理。** 正式服务应由系统生成唯一 run/case 目录，并在写入前加冲突检查或锁。

### R8. Top-K 和融合报告不是“医生认可的最佳报告”

**人话：** 目前 Top-K 主要看路由匹配、结构完整、候选之间是否一致、自己有没有前后矛盾。多个模型一起重复一个错误时，它们反而可能显得很一致。

**当前状态：仍需处理。** 系统已明确把它叫作运行性排序，不能称为临床质量排名；要变成可信排名，必须用冻结病例和医生 gold 标注验证。

### R9. 融合报告可能把多个候选的错误合在一起

**人话：** 融合不是魔法。它能整理多个候选，但没有独立真实证据时，可能把大家共同的错误写得更顺。

**当前状态：仍需处理。** 融合失败会显式显示且不会覆盖原候选；融合质量本身仍要用医生评估。

### R10. 结构化模板还没有覆盖所有部位和写法

**人话：** 系统能把很多报告拆成部位、侧别、大小、严重程度等字段，但不同医院、不同科室、中文自由写法和罕见部位不可能一次全覆盖。

**当前状态：持续完善。** 模板缺失时会退到通用结构化视图，不会让流程崩掉；但结构化分数在这些部位上会更粗糙。

### R11. “可运行”不是“已经医学验证”

**人话：** 模型能下载、能加载、能写出一段文字，只证明机器能调用它，不证明它能正确写临床报告。

**当前状态：长期必须保持。** 当前 registry 有 390 条记录，很多只是资源审计、预检查或工程 smoke；质量门阻塞的 36 条不会进入默认 production 候选。其余未完成临床验证的候选也必须带状态使用。

### R12. 模型清单和说明文档会再次漂移

**人话：** 新增或修正一个模型后，如果只改配置、不重生成导出文件和文档，网页、API、测试和说明书会说出不同数字。

**当前状态：已建立防线。** 这次从 389 修正到 390 就是实际例子。现在有统一导出和 `--check`；但以后每次改模型台账，仍必须执行它，机器不会自动替人运行命令。

### R13. OCR 或参考报告本身可能有错

**人话：** 如果原始 PDF 的 OCR 抄错了，后面的“模型和参考报告对比”就会在错误题目上评分。即使评分流程完全正确，结论也会偏。

**当前状态：仍需处理。** OCR 已有页级质量状态和阻断机制，但正式冻结 OCR 集、两次独立候选、多模态 verifier 和医生双读标注还没有全部完成。

## 六、后来确认不是实际缺陷的内容

这些项目曾被怀疑，但不应再作为开发待办：

1. `chat_completions` 不是缺失功能，已经有实现和测试。
2. LLMClient 不会在真实 provider 失败时偷偷换成 mock；mock 必须显式配置。
3. `docs/pat.txt` 没有被提交进 Git 历史。
4. 核心评测代码中的大量 `print()` 问题是误报，面向用户的 CLI 输出保留是合理的。
5. “原始医学图片一定会 base64 外发”不是独立问题，已合并到隐私配置问题。
6. `_count_workflow1` 会静默吞掉读不到的文件，这条后来被验证为不成立。
7. 旧格式迁移完全不读取 `source` 字段，这条后来被验证为不成立。
8. 关于 `chat_completions` 的另一条重复怀疑已合并，不是独立问题。

## 七、最简单的优先顺序

不考虑你要求暂缓的安全问题，下一步最值得做的是：

1. 先准备一小批真实、已确认参考答案的 X-ray、CT、MRI 病例，让医生看候选、Top-K 和融合报告。
2. 对 Yunwu、评委和 OCR 做真实重复调用，记录“同一病例多次运行是否稳定”。
3. 把网页取消按钮接到真实 worker/PID，做到点取消就真正停止后台任务。
4. 扩充 CT/MRI 序列选择规则和解剖结构模板，并用不同医院的数据验收。
5. 每次改模型台账都运行状态导出检查，避免文档、网页和 API 再次各说各话。
